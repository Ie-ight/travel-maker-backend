# 행동 기반 개인화 피드 정렬 — 기획 문서 (Draft)

사용자의 인앱 행동(북마크·별점·경로 담기)을 **측정**해, 그 사람에게 맞는 순서로 장소를 **정렬**하는 개인화 피드를 설계한다. 본 문서는 설계만 다루며 코드는 후속이다.

> **기존 자산 정책:** 현재 개발된 요소(6축 `style_vector`, 퀴즈 벡터, 기존 정렬 서비스, Celery/pgvector 인프라)는 **가능하면 재사용하고, 어떤 것도 폐기하지 않는다.** 기존 `/places/recommend`·`get_place_list`는 그대로 유지하고, 본 설계는 **새 피드 엔드포인트**로 병행 제공한다.
>
> **임베딩 문서와의 관계:** 별도 문서 [`embedding_recommendation_plan.md`](./embedding_recommendation_plan.md)는 1024D 텍스트 임베딩 노선을 다룬다. 본 문서는 **6축 기존 자산 재사용 노선**으로 출발하되, 임베딩은 §4의 "표현 레이어 전환"으로 흡수한다 — 충돌이 아니라 합성 관계다.

---

## 0. 설계의 두 직교 축

피드는 서로 **독립적으로** 고도화되는 두 축의 조합으로 동작한다. 한 축을 올려도 다른 축 코드는 바뀌지 않는다.

| 축 | 질문 | 진행 |
| --- | --- | --- |
| **A. 개인화 단계** | "이 유저에 대해 우리가 아는 게 얼마나 되나?" | **S1 인기 → S2 성향(퀴즈) → S3 행동** |
| **B. 표현 레이어** | "장소를 무슨 벡터로 표현하나?" | **6축 `style_vector` → 1024D 임베딩** (게이트 충족 시) |

- 축 A는 **유저별**로 승급한다(그 사람의 데이터가 쌓이면 다음 단계). 본 설계의 **척추**(§3).
- 축 B는 **전역적**으로 승급한다(임베딩이 충분히 쌓이면 전환). 파이프라인과 무관한 **부품 교체**(§4).
- 예: "S3 단계 유저"의 후보 생성 벡터가 6축이든 임베딩이든 §8~§10 로직은 동일하다.

## 1. 개요 / 목표

1. 유저의 행동을 측정해 **개인화된 장소 피드**를 정렬한다.
2. **새 엔드포인트**(예: `GET /api/v1/places/feed`)로 제공한다. 기존 `/places/recommend`(퀴즈 기반)·`get_place_list`(목록/검색)는 폐기 없이 유지한다.
3. 현재 6축 자산(퀴즈 `result_vector`, `PlaceFeature.style_vector`, HNSW ANN)을 최대 재사용한다.

### 현재 한계 (왜 만드는가)
- `/places/recommend`는 **퀴즈로 한 번 정해진 정적 6축 벡터**로만 정렬한다. 가입 후 아무리 북마크·리뷰를 해도 순위가 변하지 않는다.
- `get_place_list`는 `bookmark_count`/`rating_count`/`rating_avg` 기반 **전역 인기 정렬**일 뿐 개인화가 없다.
- **행동을 측정해 순위에 반영하는 피드백 루프가 통째로 없다.** 본 설계가 이 빈칸을 채운다.

## 3. [축 A · 척추] 단계적 개인화 사다리 (Progressive Personalization)

피드는 매 요청마다 **유저 데이터 성숙도로 단계를 판정**해 정렬 방식을 고른다. 이는 "예외적 폴백"이 아니라 **설계의 중심 흐름**이다.

| 단계 | 승급 조건 | 정렬 방식 | 재사용 / 신규 |
| :---: | --- | --- | --- |
| **S1 인기** | 무신호 (비로그인 또는 퀴즈X·행동X) | 전역 인기 점수 = **장소 조회수 + 리뷰수 + 북마크수**(+평점) 가중합 | `get_popular_places` 확장 (**장소 조회수 집계 추가**) |
| **S2 성향** | 퀴즈 완료 & 행동 < 10 | 퀴즈 6축 벡터 ANN | `get_places_sorted_by_vector(result_vector)` (기존 그대로) |
| **S3 행동** | 행동 ≥ 10 (`action_count`) | 행동 벡터 후보 생성 + 2단계 랭킹 | `behavior_vector`(신규) + 신규 랭킹 단계 |

### 단계 전이 규칙
- **일방향이 아니다.** 행동이 한동안 없으면(예: 최근 감쇠 후 유효 신호 < 10) S3 → S2로 강등될 수 있다. 단계는 "현재 가진 데이터"로 매번 다시 판정한다.
- **S2 → S3 경계는 부드럽게.** 행동이 N 부근일 때 갑자기 정렬이 튀지 않도록, `최종벡터 = β·behavior_vector + (1−β)·quiz_vector`로 블렌딩한다(β는 `action_count`에 따라 0→1). 행동이 적을수록 퀴즈 비중을 높게 둔다.

> **'장소 조회수'와 '사람의 클릭'은 다르다 (중요):**
> - **장소 조회수**(`Place.view_count`) = 장소가 전역적으로 몇 번 열렸나 하는 **인기 지표** → S1에 사용. 현재 미집계지만 상세 조회 시 카운터만 올리면 되니 추가가 쉽다(유저 식별 불필요).
> - **사람의 클릭·체류** = 특정 유저가 무엇을 봤나 하는 **개인 행동 신호** → S3 벡터 갱신용. 노이즈가 크고 별도 프론트 로깅이 필요해 **본 설계에서 제외**한다.
> - 정리: S1의 장소 조회수는 **유지**(전역 집계), S3의 개인 클릭은 **제외**.

## 4. [축 B] 표현 레이어 & 6축 → 임베딩 전환 (축 A와 독립)

장소를 표현하는 벡터는 **플러그인**이다. 후보 생성에 쓰는 벡터만 갈아끼우면 되고, §3·§8~§10 파이프라인은 그대로다.

- **v1 (즉시):** 6축 `style_vector` 재사용. 이미 존재하므로 바로 가동.
- **임베딩 점진 적재 (백그라운드 트랙):** 1024D `content_vector`(제목 + 분류 라벨 + 설명)를 배치 management command로 **점진 적재**. 1.5만 장 변환은 시간이 걸리므로 행동 루프 개발과 **병행**해 쌓는다.
- **전환 게이트("일정 수준 도달"):** 아래 둘을 **모두** 만족할 때만 전환을 시작한다.
  1. **커버리지:** 활성 장소의 임베딩 보유율 ≥ 90%
  2. **품질:** 오프라인 지표(§11)에서 임베딩 ANN이 6축 대비 Recall@K 우위
- **전환 방식 (하드 스위치 금지):** 점수를 블렌딩하고 비중을 **램프**로 올린다.
  ```
  score = α · cos(임베딩) + (1 − α) · cos(6축)
  ```
  α를 커버리지·지표에 따라 0 → 1로 점진 상향. 임베딩 미보유 장소는 6축으로 폴백해 **혼재 기간의 커버리지 구멍**을 막는다.

> ※ 이 축은 **유저 단계(S1/S2/S3)와 무관**하다. S2든 S3든 "벡터로 ANN을 돈다"는 동작은 같고, 그 벡터의 표현만 6축 ↔ 임베딩으로 바뀐다. 1024D 임베딩 모델·저장·인덱스 상세는 [`embedding_recommendation_plan.md`](./embedding_recommendation_plan.md)를 흡수한다.

## 5. 신규 데이터 모델 (기존 폐기 없음)

### `UserPreference` — 행동으로 갱신되는 동적 취향 (1:1 User)
| 필드 | 타입 | 설명 |
| --- | --- | --- |
| `user` | OneToOne(User) | |
| `behavior_vector` | `VECTOR(6)` | 행동 누적으로 갱신되는 취향 벡터 |
| `action_count` | `PositiveInteger` | 신뢰도·단계 판정용 유효 행동 수 |

> 퀴즈 원본 `UserTestResult.result_vector`와 **분리 보존**한다. 퀴즈 결과는 성향 설명·아바타 등에 계속 쓰이므로 동적 벡터와 섞지 않는다(폐기 없음).

### `UserActionLog` — 행동 로그 (= 벡터의 진실 원본)
| 필드 | 타입 | 설명 |
| --- | --- | --- |
| `user` | FK(User) | |
| `place` | FK(Place) | |
| `action_type` | choices | `review` / `bookmark` / `unbookmark` / `route_add` |
| `weight` | float | 산정된 기여(§6의 A 또는 W) — 배치 재계산용 |
| `processed` | bool | 멱등성: 실시간 반영 완료 여부 |
| `created_at` | datetime | 시간 감쇠·최신성 계산 기준 |

- **인덱스:** 복합 `(user, place, action_type)` (중복/조회), 부분 `WHERE processed = FALSE` (미처리 행만 빠른 스캔 — 테이블이 수백만 행이 되어도 풀스캔 방지).
- **역할:** ① 실시간 멱등성, ② 배치 전체 재계산의 원천.

### 장소 조회수 집계 (S1 인기용 — `UserActionLog`와 별개)
- `Place`에 비정규화 카운터 `view_count` 추가, 장소 상세 조회 시 `F("view_count") + 1`로 증가. **유저 식별 불필요**(per-user 행동 로깅이 아니라 전역 인기 지표).
- 쓰기 폭증이 부담이면 Redis 카운터 버퍼 → 주기 flush 옵션.

## 6. 행동 신호 측정 (Signals)

S3 벡터 갱신에 쓰는 **개인 행동 신호**다. 기본 점수 `A`를 아래와 같이 둔다 — **명시적 신호만 사용**한다(사람의 클릭·체류 같은 암묵 신호는 제외; 장소 조회수는 §3/§5의 전역 인기 지표일 뿐 여기 들어오지 않는다).

**명시적 (Explicit)** — 기존 모델에서 추출
| 행동 | A |
| --- | --- |
| 리뷰 5★ | +1.0 |
| 리뷰 4★ | +0.5 |
| 리뷰 3★ | 0.0 (중립, 미반영) |
| 리뷰 1~2★ | −0.5 (약한 밀어냄) |
| 북마크 추가 | +0.8 |
| 경로(Route)에 장소 추가 | +0.8 (강한 여행 의도) |
| 북마크 해제 | −0.1 (다녀온 뒤 해제 케이스 고려, 매우 약하게) |

> **출처:** 북마크·리뷰·경로 담기는 모두 **기존 모델에서 추출** 가능 — S3는 별도 프론트 이벤트 로깅이 필요 없다.
>
> 사람의 클릭·체류 같은 암묵 신호는 노이즈·로깅 부담으로 **본 설계 제외**(추후 확장 항목). ↔ **장소 조회수**는 개인 신호가 아니라 전역 인기 지표라 S1에서 별도로 유지(§3·§5).

## 7. 인기도(유명도) 가중치 보정

상단에 노출되는 유명한 장소를 고른 것은 취향이라기보다 **눈에 띄어서**일 확률(Position/Popularity Bias)이 높다. 반면 남들이 잘 모르는 장소를 찾아 북마크했다면 독특한 취향일 확률이 높다.

- **공식:** `P = 1 / log10(리뷰수 + 북마크수 + 10)`, 범위 `(0, 1.0]`
  - 리뷰+북마크 = 0 → P = 1.0 (기본·최댓값)
  - = 90 → P ≈ 0.5 / = 990 → P ≈ 0.33
- **공통 적용 지점 3곳:**
  1. **S3 신호 가중:** 최종 행동 가중치 `W = A × P` (인기 장소 행동의 영향 축소)
  2. **S1 인기 점수:** 과도한 부익부를 완화하는 보정항으로 활용 가능
  3. **랭킹 단계:** 과노출 장소를 소폭 다운랭크(§10)

## 8. 유저 행동 벡터 산출 (S3)

행동한 장소들의 `style_vector`를 가중·감쇠 합산해 취향 중심점(Centroid)을 만든다 — 로키오(Relevance Feedback)의 6축 적용.

> **행동과 벡터의 연결 (핵심):** 행동 자체를 임베딩하는 게 아니다. 유저가 장소를 북마크/리뷰하면 그 **장소가 이미 가진 벡터**(지금은 6축 `style_vector`, 전환 후엔 1024D `content_vector`)를 꺼내 유저 벡터에 W만큼 더할 뿐이다. 행동은 "어떤 장소를, 얼마나 긍정적으로"라는 **포인터 + 가중치**다. 그래서 표현 레이어(§4)를 바꿔도 이 식은 그대로 — 더 정밀한 장소 벡터를 더하게 될 뿐이다.

```
behavior_vector = Normalize( 시드 + Σ ( Wᵢ · decayᵢ · style_vectorᵢ ) )
```
- **시드(콜드스타트):** 퀴즈 `result_vector`. 행동이 거의 없을 때 출발점.
- **가중치:** `Wᵢ = Aᵢ × Pᵢ` (§6·§7).
- **시간 감쇠:** `decayᵢ = exp(−λ · 경과일)` — 오래된 취향의 영향을 줄인다(배치 재계산 시 적용).
- **부정 신호:** 6축은 축마다 의미가 있어 빼기 해석이 임베딩보다 안전하지만, 과도한 밀어냄을 막기 위해 빼기 계수(γ)는 더하기보다 작게 둔다.

> **안전 장치 (Zero Vector):** 부정 누적으로 L2 norm이 임계값(`1e-8`) 미만이면 0으로 나누기를 피하기 위해 중립 벡터로 리셋한다.

## 9. 하이브리드 갱신 파이프라인

사용자 경험을 해치지 않도록 벡터 계산은 백그라운드에서 처리하되, **실시간 근사 + 배치 정합**을 함께 둔다.

### 실시간 경로 (즉시 체감)
1. 유저 행동 발생 → 행동 기록 + `UserActionLog(processed=False)`를 **한 트랜잭션**으로 저장.
2. `transaction.on_commit(lambda: update_behavior_vector.delay(user_id, log_id))` — 커밋 이후에만 큐잉(커밋 전 발화 레이스 방지).
3. 워커: `UserPreference.objects.select_for_update()`로 로우 락 → **증분 가산**(기존 벡터에 `W · 장소벡터` 한 건만 더하고 정규화) → **같은 트랜잭션에서** `processed=True` 마킹. (정확도 보정은 야간 배치가 담당.)

> ⚠️ **처리 후 마킹 필수:** 처리 전에 `processed=True`로 먼저 찍으면 직후 크래시 시 해당 행동이 영구 스킵된다. 벡터 갱신과 마킹은 반드시 하나의 트랜잭션으로 원자 처리.

### 배치 경로 (진실 원본)
- Celery beat로 야간에 `UserActionLog`의 **현재 유효 행동만** 모아 `behavior_vector`를 **전체 재계산**한다.
- 정규화 벡터는 비가역이라 실시간 근사만으로는 북마크 해제·리뷰 수정이 깔끔히 반영되지 않는다. 배치 재계산이 이를 **정합**하고 누적 오차·시간 감쇠를 함께 적용한다. → `UserActionLog`가 단일 진실 원본.

## 10. 피드 정렬 상세 (S3, 2단계)

후보 생성과 랭킹을 **분리**한다. 모든 비즈니스 로직(인기 보정·최신성·태그·다양성)은 벡터가 아니라 **랭킹 단계**에 둔다.

1. **후보 생성 (recall 높게):** `get_places_sorted_by_vector(behavior_vector, …)`로 ANN 후보를 **넉넉히 과대추출**(예: top-100) ∪ 인기 일부. (태그 부스트로 순위가 크게 바뀔 수 있으니 충분히 뽑아야 후순위 후보를 놓치지 않는다.)
2. **랭킹 (precision):**
   ```
   score = cos(behavior_vector, style_vector)
         + 인기 보정(§7)
         + 최신성
         + 태그 부스트(User.tags ∩ place.tags, 상한 cap)
         − 이미 본/북마크한 장소 제외
   (+ 선택: MMR 다양성으로 비슷한 장소 연속 노출 완화)
   ```
- 태그 부스트는 상한을 둔다(예: `min(태그수 × 0.05, 0.3)`) — 코사인(−1~1)이 무의미해지는 것 방지.

## 11. 평가 및 검증

측정 가능하지 않으면 §6의 가중치 상수도, 단계·축 전환의 우위도 입증할 수 없다.

- **오프라인 (배포 전):** 유저 행동 이력을 시간순 분할 → 과거로 벡터 생성, 최근 북마크/고별점을 held-out 정답으로. 지표 `Recall@K`, `NDCG@K`(예 K=20).
  - 비교 매트릭스: **S1 vs S2 vs S3** (개인화 단계 효과), **6축 vs 임베딩** (표현 효과).
- **온라인 (배포 후):** 피드 경유 북마크율 / CTR을 A/B로 비교. **성공 기준을 사전 정의**(예: 북마크율 X% 개선)하고 미달 시 롤백.
- **모니터링:** 영벡터/NaN, 유저 벡터 norm 분포 이상치, 특정 장소 과노출.

## 12. 구현 로드맵

| 단계 | 목표 |
| :---: | --- |
| **P0** | 평가 하네스 + **S1 베이스라인**(인기순) 측정 — 이후 모든 단계의 기준선 |
| **P1** | 모델(`UserPreference`, `UserActionLog`) + 인덱스 + **장소 조회수 카운터** |

| **P4** | 피드 API 신설 — **S1→S2→S3 단계 판정** + 2단계 랭킹 + 폴백 |
| **P5** | 평가·튜닝(가중치·캡·λ·N), 이후 다중 관심사 군집화 등 고도화 |

| **P6 (게이트 충족 시)** | 6축 → 임베딩 **블렌딩 램프(α 0→1)** 전환 (§4) |

## 13. 열린 결정사항

**확정**
- ✅ 단계 승급 임계 **N = 10** (행동 10개부터 S3; 미만은 S2 퀴즈 블렌딩, 강등도 10 기준)
- ✅ `behavior_vector` 저장 = **신규 `UserPreference` 모델** (퀴즈 `UserTestResult`와 분리 보존)
- ✅ 실시간 근사 = **증분 가산** (새 행동 1건만 더하고 정규화 → 정확도는 야간 배치가 보정)
- ✅ 전환 게이트 커버리지 = **활성 장소 임베딩 ≥ 90%** (+ 오프라인 지표 우위 공통 전제)

- [ ] 시간 감쇠 `λ` 초기값 · S2→S3 블렌딩 `β` 곡선
- [ ] α 램프 스케줄 — 전환 개시 후 0→1 상향 속도
